# 下载模块重构设计（2026-03-23）

## 1. 背景与目标

当前下载模块经历多轮增量修改，出现以下问题：

- 下载链路职责分散，命令层、任务层、下载层边界不清晰
- 任务状态和实际执行线程不同步，问题排查成本高
- 进度上报链路脆弱，容易出现“状态看似正常但下载未完成”
- 旧实现包含过多历史分支（断点元数据、多模块耦合），维护成本高

本次重构目标：

1. 保证下载完整性（正确性第一）
2. 保持前端调用接口不变（`download/pause/resume/delete`）
3. 用最小复杂度实现稳定暂停/恢复
4. 提升可观测性（明确日志和状态机）

非目标（本次不做）：

- 跨应用重启的任务恢复
- 多连接并行分块加速
- 下载队列优先级调度

## 2. 设计原则

- KISS：优先顺序分段下载，避免复杂并行调度
- YAGNI：只实现当前必要能力，不保留过度抽象
- DRY：统一 HTTP 请求头、重试策略、状态上报方式
- SOLID：
  - 单一职责：Manager 管生命周期，Downloader 管网络传输，Merger 管合并
  - 依赖收敛：命令层仅做编排，不直接处理下载细节

## 3. 新架构

### 3.1 模块分工

- `downloader/manager.rs`
  - 维护任务表与控制句柄（暂停/取消）
  - 创建并启动下载任务线程
  - 统一任务状态更新与事件推送

- `downloader/chunked.rs`
  - 提供顺序 Range 下载能力
  - 支持 `.part` 文件续传（基于已下载字节）
  - 提供超时、重试、完整性校验

- `downloader/merger.rs`
  - 合并视频/音频临时文件为最终 mp4
  - 临时文件使用唯一命名，避免并发覆盖

- `commands.rs`
  - 解析请求参数
  - 调用 `DownloadManager` 启动任务或控制任务
  - 保持现有 Tauri command 接口不变

### 3.2 下载流程

1. `download` 命令接收视频列表
2. 通过 B 站接口获取视频/音频流 URL
3. `DownloadManager` 创建任务并置为 `Pending -> Downloading`
4. 对视频流执行 `download_stream_to_part`
5. 对音频流执行 `download_stream_to_part`
6. 任务状态切换为 `Merging`，调用 FFmpeg 合并
7. 合并完成后切换为 `Completed`，并清理临时目录

### 3.3 暂停/恢复/删除语义

- 暂停：设置 `paused=true`，下载循环进入短周期等待
- 恢复：设置 `paused=false`，继续从 `.part` 已下载字节处续传
- 删除：设置 `cancelled=true`，终止任务并删除临时文件（可选）

## 4. 数据与状态模型

保留现有 `TaskStatus/DownloadTask/DownloadConfig` 对前端兼容。

新增运行时控制结构（仅内存）：

- `TaskControl`
  - `paused: Arc<AtomicBool>`
  - `cancelled: Arc<AtomicBool>`

`DownloadState.active_tasks` 记录实际运行句柄，支持删除时中断。

## 5. 正确性保障

- 每个分段请求必须返回 `206`（最后越界允许 `416` 作为终止信号）
- 每段响应字节数必须大于 0
- 下载结束后校验本地文件大小与远端总大小一致
- 重试策略：指数退避（2^n 秒）+ 最大重试次数
- 超时策略：单段请求超时（配置化）

## 6. 风险与权衡

- 放弃并行分块，峰值速度可能降低，但稳定性显著提升
- 暂停/恢复为进程内能力，应用重启后不保留任务上下文
- 暂不做持久化断点元数据，避免重构首版过度复杂

## 7. 迁移计划

1. 删除旧下载模块中不再使用的代码路径（task/resume 旧实现）
2. 重写 `chunked.rs` 为简化的顺序下载实现
3. 重写 `manager.rs` 统一任务生命周期管理
4. 调整 `commands.rs` 调用新 Manager API
5. `cargo check` + 网络用例验证

## 8. 验收标准

- 指定 URL 可完整下载，最终文件可播放
- `pause/resume/delete` 行为可预测且无僵尸任务
- 下载列表状态与实际执行一致
- 同时下载多个任务不互相覆盖临时文件
